## 1. 产品概述

VocaTask 是一款基于语音的任务路由系统，专为工厂/现场场景设计。

工人用手机浏览器说一句话，AI 自动判断任务类型、分配给对应负责人，管理人员在电脑上实时看到任务并跟进处理。目标是消除手工填报任务单的流程，让现场问题在秒级内触达责任人。

## 2. 目标用户

| 角色 | 使用场景 | 使用设备 |
|------|---------|--------|
| **现场工人 / 班组长** | 发现问题、口述任务 | 手机浏览器 |
| **部门负责人 / 主管** | 查看任务、更新状态、添加处理记录 | 电脑浏览器（CRM） |

---

## 3. 核心问题

工厂现场工人发现问题时（设备故障、安全隐患、生产异常）：
- 填报纸质工单 → 慢、容易丢失
- 打电话 → 没有记录，事后扯皮
- 发微信群 → 淹没在消息里，无人负责

VocaTask 解决的是**「谁来处理」和「处理了没有」**两个问题。

## 4. 工作流程

### 工作流 A：快速任务（主流程）

```
工人举起手机
    ↓
点击录音按钮，口述任务（例："1号车间3号机床有异响，需要检修"）
    ↓
可选：拍照附上现场图片
    ↓
AI 转录 + 路由（5-10秒）
    ↓
显示结果：任务描述 / 分配给：张三（设备主管）/ 优先级：高 / 原因
    ↓
任务自动保存到 CRM
    ↓
管理人员在 CRM 上看到新任务，接手处理
```

### 工作流 B：长篇转录（辅助流程）

```
用户开始录音（现场巡检、会议记录，最长支持 90 分钟）
    ↓
提交 → 进入后台队列
    ↓
系统分片处理（每片 25 秒）+ 逐段转录 + 拼接
    ↓
完整文字稿保存，显示转录结果
```

| 功能 | 描述 | 状态 |

## 6. 技术规格

### 6.1 系统架构

```
📱 手机浏览器 (port 8010)          💻 电脑浏览器 (port 8011)
       ↓                                    ↓
┌──────────────────────────────────────────────────────┐
│                     服务器                            │
│  server.py (Flask)    crm_server.py (Flask)          │
│         ↓                      ↓                    │
│              SQLite (data/help.db)                   │
│         ↓                                           │
│  ZhipuAI API (ASR / LLM / Vision)                   │
└──────────────────────────────────────────────────────┘
```

### 6.2 技术栈

| 层 | 技术 |
|----|------|
| 后端框架 | Flask (Python 3.11+) |
| 数据库 | SQLite (WAL 模式，支持并发读写) |
| ASR 模型 | ZhipuAI GLM-ASR-2512 |
| 路由模型 | ZhipuAI GLM-4-flash（函数调用） |
| 视觉模型 | ZhipuAI GLM-4V-Flash |
| 音频处理 | ffmpeg（格式转换、分片） |
| 前端 | 原生 JavaScript（无框架） |
| 录音 API | Web MediaRecorder API |

### 6.3 数据库结构

| 表 | 用途 |
|----|------|
| `routing_history` | 所有已路由任务（转录文本、分配人、部门、优先级、方法） |
| `transcription_jobs` | 长篇转录任务状态与结果 |
| `task_attachments` | 图片附件（关联到任务） |
| `task_notes` | 任务处理笔记 |

### 6.4 API 端点

| 端点 | 方法 | 功能 |
|------|------|------|
| `/api/transcribe-and-route` | POST | 主流程：音频+图片 → 转录 → 路由 → 保存 |
| `/api/queue` | POST | 提交长篇音频到后台队列 |
| `/api/queue/status/<job_id>` | GET | 轮询长篇任务进度 |
| `/api/history` | GET | 获取任务历史列表 |
| `/api/crm/tasks` | GET | CRM 任务列表（支持过滤） |
| `/api/crm/tasks/<id>/status` | POST | 更新任务状态 |
| `/api/crm/tasks/<id>/notes` | POST | 添加处理笔记 |

## 7. 当前限制（PoC 阶段已知问题）

| 限制 | 影响 | 计划解决时间 |
|------|------|------------|
| 需要 HTTPS 才能在真机上使用麦克风 | 真机测试必须配置 HTTPS 或用 Cloudflare Tunnel | PoC 2 阶段 |
| iOS 音频格式兼容性未验证 | iOS Safari 录音格式（m4a）可能需要额外处理 | PoC 2 阶段 |
| CRM 刷新为 5 秒轮询，非真正实时 | 任务到达有最多 5 秒延迟 | 生产版 |
| 组织架构为硬编码 | 换客户需要手动改 `org_structure.py` | 生产版 |
| 无用户认证 | 任何人都能访问和操作 | 生产版 |
| SQLite 不适合高并发 | 多人同时提交时性能下降 | 生产版（迁 PostgreSQL） |
| 无推送通知 | 负责人需主动刷新 CRM 才能看到新任务 | 生产版 |

## 8. 未包含功能（当前版本 Scope 外）

以下功能在 PoC 阶段**不做**，列出供后续规划参考：

- 用户登录 / 权限管理
- 部门和人员的增删改查界面（CRM CRUD）
- 任务推送通知（微信消息 / App 推送）
- 离线录音 + 恢复后同步
- 说话人分离（区分多人对话中的发言人）
- React Native 原生 App
- PostgreSQL 数据库
- 历史任务分析报表

## 9. 部署要求

### 最低要求

| 项目 | 要求 |
|------|------|
| 服务器 | 1 台始终开机的机器（公司服务器 / VPS / 开发机） |
| Python | 3.11+ |
| ffmpeg | 需安装（音频转换） |
| 网络 | 手机和电脑都能访问服务器（见下方） |
| HTTPS | 真机麦克风必须（Cloudflare Tunnel 可自动提供） |

### 跨网访问方案

详见 `TIMELINE.md` — "网络架构与跨网访问"章节。

| 阶段 | 推荐方案 |
|------|--------|
| PoC / 演示 | Cloudflare Tunnel（免费，30分钟配置） |
| 正式生产 | VPS 云服务器 或 公司服务器+IT开通端口 |

### 启动命令

```bash
# 安装依赖
pip install flask zhipuai

# 设置 API Key
export ZHIPU_API_KEY=your-key-here

# 启动主服务（任务发送端）
python server.py          # http://localhost:8010

# 启动 CRM 服务（任务管理端）
python crm_server.py      # http://localhost:8011
```

## 10. 成功标准（PoC 验收）

| 指标 | 目标 |
|------|------|
| 语音到路由结果的端到端时长 | ≤ 10 秒 |
| 路由准确率（任务分配到正确部门） | ≥ 80%（基于测试用例集） |
| 移动端浏览器兼容性 | Android Chrome + iOS Safari 均可录音并提交 |
| CRM 任务可见延迟 | ≤ 5 秒（轮询） |
| 长篇转录（5分钟音频）处理完成时间 | ≤ 3 分钟 |
