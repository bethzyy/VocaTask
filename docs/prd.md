## 1. 产品概述

VocaTask 是一款基于语音的任务路由系统，专为工厂/现场场景设计。

工人用手机浏览器说一句话，AI 自动判断任务类型、分配给对应负责人，管理人员在电脑上实时看到任务并跟进处理。目标是消除手工填报任务单的流程，让现场问题在秒级内触达责任人。

| 角色 | 使用场景 | 使用设备 |
|------|---------|--------|
| **现场工人 / 班组长** | 发现问题、口述任务 | 手机浏览器 |
| **部门负责人 / 主管** | 查看任务、更新状态、添加处理记录 | 电脑浏览器（CRM） |

---

## 3. 核心问题

工厂现场工人发现问题时（设备故障、安全隐患、生产异常）：
- 填报纸质工单 → 慢、容易丢失
- 打电话 → 没有记录，事后扯皮
- 发微信群 → 淹没在消息里，无人负责

VocaTask 解决的是**「谁来处理」和「处理了没有」**两个问题。

## 4. 工作流程

### 工作流 A：快速任务（主流程）

```
工人举起手机
    ↓
点击录音按钮，口述任务（例："1号车间3号机床有异响，需要检修"）
    ↓
可选：拍照附上现场图片（最多5张，AI 自动描述图片内容纳入路由判断）
    ↓
AI 转录 + 路由（5-10秒）
    ↓
显示结果：任务描述 / 分配给：张三（设备主管）/ 优先级：高 / 原因
    ↓
任务自动保存到数据库
    ↓
管理人员在 CRM 上看到新任务（≤5秒刷新），接手处理
```

### 工作流 B：长篇转录（辅助流程）

```
用户开始录音（现场巡检、会议记录，最长支持 90 分钟）
    ↓
提交 → 进入后台队列
    ↓
系统自动分片（每片 25 秒）+ 逐段转录 + 拼接
    ↓
完整文字稿保存，实时显示进度（"处理中... 3/12 片段"）
```

## 6. 技术规格

### 6.1 系统架构

```
📱 手机浏览器 → http(s)://<server>:8010    💻 电脑浏览器 → http(s)://<server>:8011
        ↓                                              ↓
┌──────────────────────────────────────────────────────────────┐
│                           服务器                              │
│   server.py (Flask, port 8010)   crm_server.py (port 8011)  │
│                  ↓                        ↓                  │
│                      SQLite — data/help.db                   │
│                              ↓                               │
│                ZhipuAI API（ASR / LLM / Vision）              │
└──────────────────────────────────────────────────────────────┘
```

> 手机和电脑都是**客户端**，服务器必须有固定地址（见第 9 节部署方案）。  
> 完整技术架构详见 [architecture.md](architecture.md)。

### 6.2 技术栈

| 层 | 技术 | 说明 |
|----|------|------|
| 后端框架 | Flask (Python 3.11+) | 两个独立进程（8010 / 8011） |
| 数据库 | SQLite (WAL 模式) | 支持并发读写，PoC 阶段够用 |
| ASR 模型 | ZhipuAI GLM-ASR-2512 | 中文语音转文字 |
| 路由模型 | ZhipuAI GLM-4-flash | 函数调用，结构化输出 |
| 视觉模型 | ZhipuAI GLM-4V-Flash | 图片描述（≤80字） |
| 音频处理 | ffmpeg | 格式转换（WebM/m4a → WAV）、25秒分片 |
| 前端 | 原生 JavaScript | 无框架，Web MediaRecorder API 录音 |

### 6.3 数据库表结构

| 表 | 用途 | 关键字段 |
|----|------|---------|
| `routing_history` | 所有已路由任务 | transcribed_text, assignee, department, priority, method, status |
| `transcription_jobs` | 长篇转录任务 | status, text_result, progress |
| `task_attachments` | 图片附件 | image_path, description |
| `task_notes` | 处理笔记 | note_text, note_by |

### 6.4 主要 API 端点

| 端点 | 方法 | 功能 |
|------|------|------|
| `/api/transcribe-and-route` | POST | 主流程：音频+图片 → 转录 → 路由 → 保存 |
| `/api/queue` | POST | 提交长篇音频到后台队列 |
| `/api/queue/status/<job_id>` | GET | 轮询长篇任务进度 |
| `/api/history` | GET | 任务历史列表（路由+长篇各30条） |
| `/api/crm/tasks` | GET | CRM 任务列表（支持状态/处理人过滤） |
| `/api/crm/tasks/<id>/status` | POST | 更新任务状态 |
| `/api/crm/tasks/<id>/notes` | POST | 添加处理笔记 |
| `/api/images/<file_id>` | GET | 获取图片附件 |

## 7. 已知限制

| 限制 | 具体影响 | 计划解决阶段 |
|------|---------|------------|
| 同一 WiFi 下 HTTP 不支持麦克风 | 手机浏览器访问局域网 IP（如 192.168.x.x）时，`getUserMedia` 被浏览器安全策略拒绝；需 HTTPS 或 Cloudflare Tunnel | PoC 2 |
| 真机 HTTPS 要求 | 部署到真机测试必须配置 HTTPS（Cloudflare Tunnel 可自动解决） | PoC 2 |
| iOS 音频格式待验证 | iOS Safari 录音格式为 m4a/mp4，与 Android WebM 不同，ffmpeg 转换兼容性未实测 | PoC 2 |
| CRM 5秒轮询刷新 | 任务到达有最多 5 秒延迟，非真正实时推送 | 生产版 |
| 组织架构硬编码 | 换客户需手动修改 `core/org_structure.py` | 生产版 |
| 无用户认证 | 任何能访问服务器的人都可操作 | 生产版 |
| SQLite 并发限制 | 多人同时提交时性能下降 | 生产版（迁 PostgreSQL） |
| 无推送通知 | 负责人须主动刷新 CRM | 生产版 |

## 9. 部署方案

### 9.1 运行环境要求

| 项目 | 要求 |
|------|------|
| 服务器 | 1 台始终开机的机器（公司服务器 / VPS / 开发机） |
| Python | 3.11+ |
| ffmpeg | 必须安装（音频格式转换） |
| HTTPS | 真机麦克风调用必须（见下方方案选择） |

### 9.2 环境变量配置（.env 文件）

项目使用 `.env` 文件管理环境变量，`.env` 已加入 `.gitignore` 不进代码库。

```bash
# .env 文件（复制 .env.example 后填入真实值）
ZHIPU_API_KEY=your-zhipu-api-key
CLAUDE_API_KEY=your-claude-api-key        # 可选，测试用
FFMPEG_PATH=/usr/bin/ffmpeg               # Linux 服务器通常为此路径
                                          # Windows 示例：C:\tools\ffmpeg.exe
```

### 9.3 网络访问方案

手机和电脑需要能访问同一台服务器。根据部署阶段选择方案：

| | 方案 A：VPS 云服务器 | 方案 B：公司服务器 + Cloudflare Tunnel | 方案 C：公司服务器 + IT 开通 |
|--|---|---|---|
| **适用阶段** | 生产 | **仅 PoC / 演示** | 生产 |
| **费用** | 60–200 元/月 | 免费 | 无额外费用 |
| **搭建时间** | 3–4 小时 | 30 分钟 | 3–5 天 |
| **数据位置** | 云端（公司外） | 公司服务器 | 公司服务器 |
| **数据经过第三方** | 否 | 是（Cloudflare，加密） | 否 |
| **需要找 IT** | 否 | 否 | 是 |
| **自动 HTTPS** | 否（需配置） | ✅ 是 | 否（需配置） |

> 详细搭建步骤见 `TIMELINE.md` — "部署：网络架构与托管方案"章节。

### 9.4 启动命令

```bash
# 1. 复制并填写环境变量
cp .env.example .env
# 编辑 .env，填入 ZHIPU_API_KEY 和 FFMPEG_PATH

# 2. 安装依赖
pip install flask zhipuai python-dotenv

# 3. 启动主服务（任务发送端）
python server.py          # → http://0.0.0.0:8010

# 4. 启动 CRM 服务（任务管理端，另开终端）
python crm_server.py      # → http://0.0.0.0:8011
```

| 指标 | 目标 |
|------|------|
| 语音到路由结果端到端时长 | ≤ 10 秒 |

## 11. 与原始需求对比

原始设计文档（`request.md`）定义了长期愿景，当前 PoC 用更快的技术路线验证核心流程。

| 维度 | 原始设计（request.md） | 当前 PoC (v0.2) |
|------|---------------------|----------------|
| 移动端 | React Native App | 手机浏览器 |
| 主 AI | Claude Haiku 4.5 | ZhipuAI GLM-4-flash |
| 备用 AI | Gemma 4 E2B（本地推理） | 关键词规则匹配 |
| ASR | Claude API | ZhipuAI GLM-ASR-2512 |
| 说话人分离 | Gemma 4 E2B | 未实现 |
| 数据库 | PostgreSQL | SQLite |
| 离线支持 | 本地队列 + 同步 | 未实现 |
| 推送通知 | Firebase / APNs | 未实现 |

> PoC 验证通过后，按 `TIMELINE.md` 路线图逐步向原始设计靠拢。
